레스트 API: 하이퍼텍스트를 포함한 self-descriptive한 메세지의 uniform interface를 통해 리소스에 접근하는 API(제약조건을 따라야함)

<셀프디스크립티브 메세지>

메세지는 스스로를 설명해야한다(메세지는언제나 해석이 가능해야함)

1.상태 명시(HTTP status)
2.어떤 형식으로 작성된 정보인지 명시(컨텐츠 타입이 있어야함)

IANA에 미디어 타입 등록 -> 메세지의 의미를 온전히 해석 가능
3.링크 헤더에 명세를 확인할수 있는 링크를 넣음 -> rel=profile을 해서 id가 뭐고 title이 뭔지 의미를 정의한 명세가 있으면 됨

<헤이트오스>

애플리케이션 상태는 Hyperlink를 이용해서 전이가 되어야한다 -> 접근 가능한 자원들을 하이퍼링크로 통해 알려줘야함
->  a 태그가 존재하고 그것을 통해 다음 상태가 결정되는 HTML은 HATEOAS를 만족함


다음상태로 전이할 링크가 있어야함
목적지가 있어야함

다양한 방법으로 하이퍼링크를 표현함
(link태그를 이용함)


<독립적 진화>
서버와 클라가 독립적으로 진화

레스트는 웹의 독립적진화를 위해 만들어짐
-> 웹을 망가뜨리는거 없이 http를 어떻게 향상 시킬것인가 <- REST를 만들게 된 계기

서버의 기능이 변경되도 클라이언트를 업데이트할 필요가 없다

레스트 조건:유니폼 인터페이스를 만족해야한다 -> 유니폼 인터페이스(한정된 인터페이스: GET, POST, PUT, DELETE 같은 방식)로 
리소스에 접근 하는 방식 -> URI주소가 길어지는것을 줄임(URI에는 리소스만 표현하면 되기 때문)


웹(레스트를 매우 만족)
웹페이지를 변경해도 웹브라우저를 업데이트할 필요가 없음(그 반대도 마찬가지)
http 명세, html 명세가 변경되도 웹은 잘 작동함
